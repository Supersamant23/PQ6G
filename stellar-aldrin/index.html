Problem Identification

The Quantum Threat

Current encryption standards — RSA-2048, ECC, DH — rely on the computational hardness of integer factorization and discrete logarithms. Quantum computers running Shor's Algorithm will break these in polynomial time.

"Harvest Now, Decrypt Later"

Nation-state adversaries are already exfiltrating encrypted 6G traffic today, stockpiling it for decryption once cryptographically relevant quantum computers arrive (estimated 2030–2035).

Project Objectives

Primary Goal

Implement a Hybrid Security Layer for 6G networks that unifies post-quantum cryptography (PQC) with real-time, stream-based intrusion detection.

  • Kyber-768 (FIPS 203): Quantum-resistant Key Encapsulation Mechanism based on Module-LWE.
  • Dilithium-3 (FIPS 204): Quantum-resistant Digital Signatures based on M-LWE / M-SIS.
  • AES-256-GCM: Symmetric-key data encryption post-handshake for throughput performance.
  • Real-time IDS: Apache Flink streaming analytics to detect FLOOD, STEALTH, and BURST attacks in 10-second tumbling windows.

Security Outcomes (CIA Triad)

Confidentiality: Kyber-768 KEM protects session keys against quantum-capable adversaries.
Integrity: Dilithium-3 signed handshakes prevent MITM and forgery.
Availability: Real-time DDoS detection (Flink) + ML classification (Spark) maintain service uptime.

PQC Handshake Protocol

NIST Standardized Algorithms

Each network flow performs a real Kyber key exchange and Dilithium signature via liboqs 0.10.1 compiled into the NS-3 simulator.

Client → ClientHello
Server → ServerHello (Kyber PubKey + Dilithium Sig)
// Client verifies signature & encapsulates Client → KeyExchange (Kyber Ciphertext)
// Both derive shared secret Both → Switch to AES-256-GCM

Why Lattice-Based Crypto?

  • Kyber (M-LWE): Public key ~1,184 B, ciphertext ~1,088 B — compact enough for 6G mobile devices.
  • Dilithium (M-LWE/SIS): Strong EUF-CMA authentication, sigs ~3,293 B — prevents MITM during handshake.
  • AES-GCM: Post-handshake symmetric encryption. Fast. Proven. No quantum weakness for 256-bit keys.

Flink Real-Time Pipeline

Attack Detection Engine

Apache Flink 1.18 consumes flow events from Kafka, applies rule-based heuristic classification, and emits alerts + aggregated statistics in 10-second tumbling windows.

def detectAttackType: String = { if (flow_bytes_s > 2M || flow_packets_s > 4K) "FLOOD" else if (flow_bytes_s > 500K && iat_std > iat_mean*0.3) "BURST" else if (flow_bytes_s < 5K && pkt_len_mean < 100) "STEALTH" else "Normal" }
  • Severity Levels: CRITICAL · HIGH · MEDIUM · LOW — based on bytes/sec and packets/sec thresholds.
  • Dual Sink: Alerts → Kafka (security-alerts) + HDFS file sink for long-term retention.

Spark MLlib Classification

Random Forest Classifier

Spark 3.5.1 trains a multiclass Random Forest (Normal / FLOOD / STEALTH / BURST) on labeled ground-truth generated by the NS-3 simulator.

  • Trees: 100  |  Max Depth: 10  |  Seed: 42
  • Split: 80% train · 20% test
  • 15 Features: Flow bytes/s, packets/s, IAT (mean, std, max, min), pkt length stats, duration, fwd packets, KEM time, SIG sign/verify time.
  • Pipeline: StringIndexer → VectorAssembler → RF → IndexToString

Structured Streaming Classifier

After training, the model is loaded in Spark Structured Streaming mode to classify new flows from Kafka in real-time. Predictions are written to HDFS.

Dual Detection Strategy

Layer 1 — Flink (Rule-Based): Threshold heuristics for immediate low-latency alerts.
Layer 2 — Spark (ML-Based): Random Forest for nuanced multiclass classification with feature importance.

NS-3 Network Simulation

Simulation Design

NS-3.41 simulates a star topology with a central router, modeling 6G point-to-point links at 10 Gbps with 0.1 ms latency. Each flow executes a real PQC handshake via liboqs.

  • Topology: Star — sources → router → destinations (1 pair per flow).
  • Default Config: 20 Normal + 3 Flood + 3 Stealth + 2 Burst flows, 30s duration.
  • Attack Models: OnOff applications with calibrated rates (50 Mbps flood, 100 Kbps stealth, 100 Mbps burst).
  • FlowMonitor: Captures per-flow stats — bytes, packets, IAT, delay, packet lengths.

Per-Flow PQC Execution

Not simulated — actually executed. Each flow calls OQS_KEM_keypair, OQS_KEM_encaps, OQS_KEM_decaps, and OQS_SIG_sign / OQS_SIG_verify, then logs the overhead in milliseconds.

// C++ — pqg6-sim.cc OQS_KEM *kem = OQS_KEM_new(OQS_KEM_alg_kyber_768); OQS_KEM_keypair(kem, public_key, secret_key); OQS_KEM_encaps(kem, ciphertext, shared_secret_enc, public_key); OQS_KEM_decaps(kem, shared_secret_dec, ciphertext, secret_key); // Verify shared secrets match bool ok = memcmp(shared_secret_enc, shared_secret_dec, len) == 0;

Live Dashboard

Real-Time Monitoring

A Node.js + Express backend subscribes to Kafka topics via KafkaJS and broadcasts events over WebSocket to the browser UI in real-time.

  • WebSocket Push: Every Kafka message (flows, alerts, stats) is broadcast to all connected clients instantly.
  • REST API: /api/alerts · /api/stats · /api/health · /api/spark-results
  • Service Health: Polls Kafka, HDFS, Flink JobManager, and reports container status.
  • Simulation Trigger: POST /api/simulate launches NS-3 with custom parameters via Docker Engine API (socket-level).
